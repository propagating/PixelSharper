# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

PixelSharper is a C# port of Javidx9's [olcPixelGameEngine](https://github.com/OneLoneCoder/olcPixelGameEngine) (a single-header C++ software-rendering game engine). The goal is a faithful translation that is restructured into idiomatic, file-per-type C#, leans on .NET hardware-accelerated generic math / intrinsics for performance, and is backed by unit tests. The long-term consumer is a planned NES emulator ("NESharp"). Licensed OLC-3.

**The C++ source of truth lives in the repo** and is the spec for porting:
- `olcPixelGameEngine.h` (~8100 lines, at repo root and duplicated in `PixelSharper.Core/`) — the engine being translated.
- `olcExampleProgram.cpp` — reference usage / behavior.

When porting a feature, find the corresponding C++ in `olcPixelGameEngine.h` and translate it, splitting nested C++ classes (`olc::Sprite`, `olc::Decal`, `olc::Renderer`, etc.) into their own C# files under the namespace layout below.

## Commands

```powershell
dotnet build PixelSharper.sln                      # build everything
dotnet run --project PixelSharper.Core             # run the demo Main (see caveat below)
dotnet test                                         # run all NUnit tests
dotnet test --filter "FullyQualifiedName~SpriteTests"          # one fixture
dotnet test --filter "Name=Constructor_Default_ShouldSetWidthAndHeightToZero"  # one test
```

Target framework is **net7.0**, `LangVersion 11`, `Nullable` and `ImplicitUsings` enabled. .NET 7 is required because the port relies on **static abstract interface members / generic math** (`INumber<T>`), which do not exist in earlier versions. (Stray `obj/Debug/net6.0` artifacts exist but the projects target net7.0.)

`PixelSharper.Core` is currently an `Exe`. `Program.Main` is a throwaway harness with **hardcoded absolute paths** (`E:\Projects\...`, `C:\Users\Ryan\Desktop\...`) used to exercise the resource pack code — it is not a real entry point and will fail on other machines. Don't treat it as stable; tests are the reliable way to validate behavior.

## Architecture & conventions

**Two projects:** `PixelSharper.Core` (engine) and `PixelSharperTests` (NUnit + coverlet). One type per file, organized by role rather than mirroring the flat C++ header:

- `Components/` — engine objects: `Pixel`, `Sprite`, `SpritePatch`, `Decal`, `DecalPatch`, `Renderer`.
- `Types/` — math/value types: `Vector2d<T>`, `Vertex`, `LayerDesc`, `GpuTask`.
- `Enums/` — one enum per file (`DecalMode`, `CullMode`, `FileReadCode`, `Mouse`, `KeyPress`, …).
- `Resources/` — asset/pack I/O: `ResourcePack`, `ResourceFile`, `ResourceBuffer`, `ImageLoader`.
- `Actions/`, `Instance/` — `Renderable`, `DecalInstance`. **Note:** the folder layout and namespaces are not fully in sync (e.g. `Instance/DecalInstance.cs` declares `namespace PixelSharper.Core.Actions`). Match the *namespace* a type already uses, not its folder, until this is reconciled.
- `PixelGameEngine` (`PixelGameEngine.cs`, formerly `CoreEngine.cs`) — abstract base the user subclasses, mirroring `olc::PixelGameEngine`. Override `OnCreate()` / `OnUpdate(float)`; call `Construct(...)` then `Start()`. See `PixelSharperEngine` in `Program.cs` for the pattern.

**Porting idioms established in the codebase — follow these:**
- **Generic numeric types via `INumber<T>`.** `Vector2d<T>` (`Types/Vector2d.cs`) is the canonical example: a `struct` constrained `where T : struct, INumber<T>, ...`, with operators and helpers marked `[MethodImpl(MethodImplOptions.AggressiveInlining)]`, using `T.CreateChecked(...)` to convert between numeric types. New math types should follow this shape rather than hardcoding `int`/`float`.
- **`Vector2d` operators are same-type only.** Operators (`+ - * /`, scalar `*`/`/`) exist only for matching `T` — there are intentionally **no** typed (`float`/`double`/`int`) scalar or cross-type vector overloads, because a typed overload is ambiguous with the generic `T` one when `T` equals that type (e.g. `Vector2d<float> * float`). For mixed-type math, convert explicitly first with **`v.As<TOut>()`** (e.g. `vfloat / visize.As<float>()`). (The old cross-type `/` overloads also silently dropped the Y component — a bug now removed.)
- **C++ unions → explicit struct layout.** `Pixel` (`Components/Pixel.cs`) is `[StructLayout(LayoutKind.Explicit, Pack = 1, Size = 5)]` with overlapping `[FieldOffset]` fields to reproduce the `olc::Pixel` `union { uint32 n; struct{r,g,b,a} }`. Reuse this technique for other unioned C++ types; keep packing/offsets exact.
- **Static config object.** Engine-wide constants come from `PixelConfiguration` (a `readonly record struct` in `CoreConfiguration.cs`), exposed as `PixelGameEngine.Configuration`.
- **Renderer is an abstract device interface, backed by OpenTK.** `Renderer` (`Components/Renderer.cs`) is the abstract GL device; `Renderers/RendererOgl10.cs` is the first concrete backend — a faithful port of olc's `Renderer_OGL10` (fixed-function immediate mode) using **OpenTK 4.x** (`OpenTK.Graphics.OpenGL` legacy GL for immediate mode; `IGraphicsContext` from `OpenTK.Windowing.Common` for swap/make-current). A future OGL33 shader/VBO renderer can sit alongside it behind the same abstract base. The live device is reached via the static `Renderer.Active` (mirrors olc's namespace-global `renderer`). `ImageLoader` is similarly an abstract base with no concrete backend yet.
- **Platform is the OpenTK windowing sibling.** `Components/Platform.cs` is the abstract OS/window/event-loop base (port of olc's `Platform`); `Platforms/PlatformOpenTK.cs` implements it on an OpenTK `NativeWindow`. olc splits the OS event loop (main thread) from the GL context (engine thread); GLFW is single-threaded-friendly, so **we collapse to one thread** — the engine loop pumps events via `HandleSystemEvent` each frame and `StartSystemEventLoop` is a no-op. `CreateWindowPane` builds the window with a **compatibility** GL profile (required for OGL10 immediate mode — a Core profile rejects `glBegin/glEnd`); `CreateGraphics` makes the context current and hands it to `Renderer.Active.CreateDevice`. The `NativeWindow` is exposed via `PlatformOpenTK.Window` so the (not-yet-written) engine loop can query client size / detect `IsExiting`.
- **The engine loop runs end-to-end.** `PixelGameEngine` (`PixelGameEngine.cs`) now implements the full lifecycle: `Construct` sets screen/pixel/window sizing; `Start` wires `PlatformOpenTK` + `RendererOgl10` (sets `Renderer.Active`/`Renderer.PtrPGE`/`Platform.PtrPGE`), creates the window + GL context, and runs a single-threaded core loop (`CoreUpdate`) until `PlatformOpenTK.ShouldClose`. Each frame: pump events → user `OnUpdate` (which draws into the layer-0 draw-target `Sprite` via `Draw`/`Clear`) → upload dirty layer textures (`Decal.Update`) → `DrawLayerQuad` → `DisplayFrame`; FPS goes in the window title. `Program.cs` is a runnable demo (animated gradient) — `dotnet run --project PixelSharper.Core` opens a window. **GLFW context note:** immediate-mode OGL10 requires requesting GL **2.1 + `ContextProfile.Any` + `ContextFlags.Default`** (a profile hint is only legal for ≥3.2, and forward-compat only for ≥3.0) — see `PlatformOpenTK.CreateWindowPane`.
- **Input is wired (pull model).** Each frame `CoreUpdate` polls `PlatformOpenTK` (which reads OpenTK's snapshotted `KeyboardState`/`MouseState`), fills the raw new-state arrays, then `ScanHardware` derives `Pressed`/`Released`/`Held` edges exactly like olc. Query via `GetKey(KeyPress)`, `GetMouse(int)`, `GetMouseX/Y`, `GetMousePos`, `GetMouseWheel`, `IsFocused`. The `KeyPress`→GLFW `Keys` map lives in `PlatformOpenTK.BuildKeyMap` (SHIFT/CTRL match either side; OEM_8 has no GLFW equivalent). `HardwareButton` state is held in **arrays** (not `List`) so element-field writes mutate in place. Note: `PlatformOpenTK` imports GLFW `Keys`/`MouseButton` via `using` aliases because that namespace also defines a `Platform` type that collides with our base class.
- **Software draw primitives are ported.** `PixelGameEngine` has `Draw`, `Clear`, `DrawLine` (with the dashed bit-`pattern` and Cohen–Sutherland `ClipLineToDrawTarget`), `DrawCircle`/`FillCircle`, `DrawRect`/`FillRect`, `DrawTriangle`/`FillTriangle`, and `DrawSprite`/`DrawPartialSprite` (scale + `SpriteMirrorMode` flip), each with `Vector2d<int>` overloads. `FillTriangle` is a faithful `goto`-for-`goto` port of olc's scanline fill. All write into the current draw-target `Sprite` via `Draw` (no renderer dependency).
- **Text is ported.** `ConstructFontSheet` (called from `PrepareEngine`, needs the GL context) decodes olc's embedded base64-ish run data into a 128×48 8×8-glyph font `Sprite` and builds the proportional spacing table. `DrawString`/`DrawStringProp` (mono + proportional, scalable) and `GetTextSize`/`GetTextSizeProp` read the font-sprite pixels — software, no renderer dependency. They temporarily switch the pixel mode to `Mask`/`Alpha` for transparency and restore it.
- **GPU decal path is ported.** `PixelGameEngine` queues `DecalInstance`s (and `GPUTask`s for rotation) into the current layer's lists, which `CoreUpdate` flushes to the renderer. Implemented: `DrawDecal`, `DrawPartialDecal` (both overloads), `DrawExplicitDecal`, `DrawPolygonDecal` (tint / per-vertex / depth variants), `DrawRotatedDecal` (submitted as a `GPUTask`), `DrawPartialRotatedDecal`, `DrawWarpedDecal`/`DrawPartialWarpedDecal` (projective), the colour-quad decals (`DrawLineDecal`, `DrawRectDecal`, `FillRectDecal`, `GradientFillRectDecal`, `FillTriangleDecal`, `GradientTriangleDecal`), and the string decals (`DrawStringDecal`/`DrawStringPropDecal`/rotated variants). Engine-level `SetDecalMode`/`SetDecalStructure` set `_decalMode`/`_decalStructure` used when building instances. These methods do their `vf2d` math with scalar `.X/.Y` (predating the `Vector2d` operator cleanup; still valid). Added the faithful `Pixel * Pixel` (normalised component multiply) that the decal tinting needs.
- **Textured triangles & patches are ported.** `FillTexturedTriangle` (software gouraud + textured scanline fill, a faithful port of olc's triangles.c) and `FillTexturedPolygon` (List/Strip/Fan), plus the patch overloads `DrawSprite(Vector2d<float>, SpritePatch, scale)` (→ `FillTexturedPolygon`) and `DrawDecal(Vector2d<float>, DecalPatch, scale)` (→ `DrawPolygonDecal`). `DecalPatch.Coordinates` is now allocated as a 4-element array (was an uninitialised null — `Decal.Patch` NRE'd on first use).
- **HW3D, dropped-files, and the last decal overload are done.** `HW3D_Projection`/`HW3D_EnableDepthTest`/`HW3D_SetCullMode`/`HW3D_DrawObject`/`HW3D_DrawLine`/`HW3D_DrawLineBox` build depth/cull-enabled `GPUTask`s. Dropped files: `PlatformOpenTK` subscribes to OpenTK's `FileDrop`; `CoreUpdate` consumes them (valid one frame, then cleared) and exposes `GetDroppedFiles()`/`GetDroppedFilesPoint()`. The `DrawPolygonDecal(decal, pos, uv, colours, tint)` convenience overload (per-vertex colours × tint) is added. The olc 2D/3D PixelGameEngine API surface is now essentially complete.
- **Window-resize works.** `CoreUpdate` polls `PlatformOpenTK.ClientSize` each frame (pull-model stand-in for olc's OS resize callback); on a change it calls `UpdateWindowSize`, which recomputes the letterbox viewport (`UpdateViewport`) — the per-frame `renderer.UpdateViewport(ViewPos, ViewSize)` then applies it. In real-window mode it also flags `_resizeRequested`, handled after `DisplayFrame` by `SetScreenSize` (rebuilds layer draw targets at the new size), faithful to olc. Verified: a forced 700×500 on the 256×240 (4×) screen produced `viewPos=83,0 viewSize=533×500` (aspect preserved, centered).
- `LayerDesc` is a **class** (not a struct) because the loop mutates layers in place in a `List<LayerDesc>`.

## Extensions & utilities (olc PGEX / UTIL ports)

The olc source headers to port live in the top-level **`extensions/`** (11× `olcPGEX_*`) and **`utilities/`** (8× `olcUTIL_*`) folders. PGEX-extension ports go in **`PixelSharper.Core/Extensions/`** (namespace `PixelSharper.Core.Extensions`); standalone utility ports go in **`Utilities/`** (`PixelSharper.Core.Utilities`); shared math types go in `Types/`.

- **`PGEX` base** (`Extensions/PGEX.cs`) — port of `olc::PGEX`, the base for all extensions. Construct with `hook:true` to auto-register so the engine calls its lifecycle hooks. The engine sets the static `PGEX.Pge` in its constructor, keeps a `_extensions` list (`RegisterExtension`), and calls `OnBeforeUserCreate`/`OnAfterUserCreate` around `OnCreate`, and `OnBeforeUserUpdate(ref elapsedTime)`/`OnAfterUserUpdate` around `OnUpdate` in `CoreUpdate`. `OnBeforeUserUpdate` returning `true` **blocks** the user frame (how `SplashScreen` plays first).
- **Extensions ported:** `SplashScreen` (auto-hook animated OLC splash), `Wireframe` (`Wire.Model` hierarchical models + mesh factories + `DrawModel`, with both a plain-engine and a `TransformedView`-aware overload), `GFX2D` (affine `Transform2D` + back-sampling `DrawSprite`), and **`TransformedView`** (`Extensions/TransformedView.cs` — the pan/zoom camera: world↔screen transforms, `HandlePanAndZoom`, and `World*` wrappers over every engine draw; plus `TileTransformedView`). `Matrix2D` (`Types/Matrix2D.cs`) is a shared 3×3 transform used by Wireframe.
- **Utilities ported:** `Palette` (`Utilities/Palette.cs` — interpolated colour palette, continuous `Sample`/discrete `Index` LUT; unit-tested in `PaletteTests`).
- **Incidental fix:** `Vector2d.Clamp` was only applying the lower bound to X and upper to Y — fixed to clamp both components to `[min,max]` (TransformedView's zoom-clamp needs it; the existing test still passes).
- **Porting note:** olc extensions are duck-typed C++ templates / header-only; in C# use interfaces or concrete `PixelGameEngine`/view types. Watch for the removed `Vector2d<float>` scalar operators — use scalar `.X/.Y` math (the extensions do a lot of `vf2d` arithmetic).
- **Decal/GPU-task allocations are pooled.** `DecalInstance` and `GPUTask` are **classes** with a `Reset()` (clears their vertex `List`s, keeping capacity). Each layer's `VecDecalInstance`/`VecGPUTasks` double as **persistent object pools** with a live-count field (`DecalInstanceCount`/`GpuTaskCount`); submit methods rent via `NextDecalInstance()`/`NextGpuTask()` (recycle or grow), and `CoreUpdate` resets the count each frame instead of clearing the list — so the objects and their lists are reused, not re-allocated. **Gotcha:** `GPUTask.Mvp` is an owned 16-float array; `HW3D_*` must `Array.Copy` into it (never replace the reference), so `Reset()` can restore identity in place without corrupting a caller's matrix. (Measured: pooling dropped steady-state allocation from ~1.2 MB/s to ~280 KB/s on the demo; combined with the 4-byte `Pixel`/zero-copy upload, total is ~36 MB/s → ~280 KB/s with 0 gen0 GCs.)
- **Pixel is exactly 4 bytes and blittable** (`[StructLayout(Explicit, Size=4)]`, union of `N` and `Red/Green/Blue/Alpha`) — matching olc (its `Mode` is an enum *type*, not a per-pixel field). This is load-bearing for performance: `RendererOgl10.UpdateTexture`/`ReadTexture` upload **directly** from the `List<Pixel>` backing array via `CollectionsMarshal.AsSpan` + `MemoryMarshal.GetReference` — **zero allocation, zero copy** per frame. (Measured: this removed ~35 MB/s of allocation and ~11 gen0 GCs/sec for a 256×240 screen at 144 fps — the old code allocated a fresh `uint[w*h]` per `UpdateTexture`.) Likewise `Clear` uses a vectorised span `Fill`. **Keep `Pixel` blittable** — adding a managed field or changing its size would break the direct upload.

**Testing (NUnit):** Arrange/Act/Assert with comment markers, `[TestFixture]`/`[Test]`/`[SetUp]`. External dependencies are injected via static fields and replaced with mocks in `Setup()` — e.g. `Sprite.ImageLoader = new MockImageLoader();`. When porting a type that touches I/O, give it the same seam (a virtual/abstract collaborator on a static field) so it stays testable. Add a `<Type>Tests.cs` fixture per ported type.

**`LinqPad/`** holds `.linq` scratch scripts used to prototype tricky translations (AES transform, struct layout, vector math, string sizing) before committing them into the engine. Use it for the same purpose; it is not part of the build.
